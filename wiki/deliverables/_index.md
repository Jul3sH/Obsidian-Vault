# Deliverables — Backlog

A deliverable has one of three types (`type:` in frontmatter):

| Type | Meaning | How it's created |
|------|---------|-------------------|
| `user-story` | User-facing value | `/define-user-story` |
| `enabler` | Technical work (Exploration/Architecture/Infrastructure/Compliance) | `/define-enabler` |
| `task` | Anything else — generic deliverable | `/define-task` |

Prioritisation (MoSCoW) happens during `/timebox-plan`, not here — this is the unprioritised backlog.

## Admission Criteria

Every item in this backlog has passed three gates:

| Gate | What it proves | Where it's captured |
|------|---------------|-------------------|
| **Scoped** | Work is defined enough to estimate | Acceptance criteria / success criteria exist |
| **Sized** | Effort is understood | Story points assigned |
| **Value-linked** | Delivers against a validated objective | Traces to an active Project Outcome/Output — or has standalone value rationale |

Items linked to an active Project but not yet ready live on that Project's
own page under its `## Funnel` section. Standalone ideas — not linked to any
Project — become XXS Project candidates instead; see
[[../projects/funnel|Project Funnel]].
