# Morning Skill — Daily Plan Generator

Runs at 6 AM HKT to generate a daily plan. Fetches the current sprint from
Jira to get live story statuses, cross-references wiki/projects/_index.md for
WSJF scores, maps remaining work to your ideal day template, and populates
Google Calendar with the allocated work blocks.

**Output:** Google Calendar events for today's remaining sprint stories,
WSJF-ordered, fit to your adjusted focus capacity.

---

## Ideal Day Template

**Disruption Detection:**
- Ask: "Are there any disruptions to your ideal day profile?"
- Disruptions may include: running, gym sessions, lunch meetings, or other deviations
- Parse disruptions to calculate adjusted focus capacity
- Default (no disruptions): 8h total focus (7 AM–12 PM + 1–4 PM)

**Focus Blocks (managed by this skill):**
1. **Early morning:** 7–8 AM (1h) — "eat the frog" hardest task
   - Skipped only if a morning disruption (e.g., running, gym) occupies this slot
2. **Morning core:** 7–12 PM (5h, or 8 AM–12 PM if running)
3. **Afternoon:** 1–4 PM (3h)

**After 4 PM:** Flexible. Only schedule additional work if the 8h target wasn't hit
due to calendar disruptions or confirmed user disruptions.

## Step-by-Step Process

### Step 0 — Fetch Live Sprint Status from Jira

**Purpose:** Get ground truth on what's actually remaining before planning the day.
The wiki may be stale; Jira always reflects the latest story statuses.

**Connection:**
- Site: agileict.atlassian.net
- Cloud ID: `bbbb75d2-e2e4-44fe-a329-e506d1128c29`

**JQL query:**
```
project = BWS AND sprint in openSprints() ORDER BY priority ASC
```

**Extract for each story:**
- Key (BWS-N)
- Summary (title)
- Status (To Do / In Progress / Done)
- Story Points (`customfield_10016`) = hours estimate
- Priority
- Labels (to extract `wsjf-X.X` label if present)

**Filter:**
- Exclude stories with status = **Done** — these are finished and don't need to be planned
- Include: To Do, In Progress (in progress stories go to the top — finish what's started)

**If no open sprint exists:**
- Display: "No active sprint found in Jira. Run `/sprint-plan` to create one."
- Continue using wiki WSJF queue as fallback (Step 2)

**If sprint exists but all stories are Done:**
- Display: "All sprint stories are complete — great work! No Jira stories to plan today."
- Continue to Step 1 (disruptions), skip Steps 2 and 5 unless wiki queue has queued projects

**Output:** `jira_stories` list — key, title, status, hours, priority, wsjf_label

### Step 1 — Ask About Disruptions to Ideal Day

**Input:** Ask the user: "Are there any disruptions to your ideal day profile? (e.g., running, gym, lunch meeting, or 'none')"

**Logic:**
Parse the user's response to identify disruptions and calculate adjusted capacity:

- **If "none":** Standard capacity = **8h total** (7 AM–12 PM = 5h + 1–4 PM = 3h)
- **If running mentioned:** Morning shifts to 8–12 PM (4h) + 1–4 PM (3h) = **7h total**
- **If gym + duration stated:** Subtract gym duration from morning block (e.g., 1h gym → morning becomes 4h, total = 7h)
- **If lunch meeting with different timing:** Adjust the 12–1 PM lunch block; recalculate morning/afternoon availability
- **If multiple disruptions:** Add up the time impact and recalculate total capacity

**Key assumption:** The user typically runs or does one activity. If multiple or overlapping disruptions, ask for clarification: "So you're running (1h) + lunch meeting (1h at 11:30–12:30)? That's 2h of disruption — should I plan for 6h of focus?"

**Output:**
- Store disruptions_desc = user's text response
- Calculate adjusted_capacity_hours (8h, 7h, 6h, etc.)
- Determine start time for morning block (7 AM by default, 8 AM if running/early gym, etc.)
- Pass both to Step 5 for project allocation.

---

### Step 2 — Cross-Reference WSJF Scores from Wiki

**Purpose:** Jira has live status; the wiki has WSJF scores. This step merges them.

**Source:** Read `wiki/projects/_index.md`

**For each story from Step 0's `jira_stories` list:**
- Look up the matching wiki project by Jira key (e.g., BWS-2)
- Extract WSJF score, t-shirt size, hours estimate (use wiki hours if Jira story points are missing)
- If no wiki match found: use Jira priority as a proxy (Highest=5, High=4, Medium=3, Low=2, Lowest=1) and flag it

**Ordering:**
1. **In Progress** stories first — always finish what's started
2. Then remaining **To Do** stories ordered by WSJF score descending

**If no open sprint (fallback from Step 0):**
- Read full wiki queue as normal
- Filter: status = "queued" or "active", exclude "completed"

**Output:** Merged and ordered working list:
```
→ IN PROGRESS
1. BWS-2  Integrate Platform & Send   WSJF 4.5  S  8h  In Progress ← finish first

→ TO DO (WSJF order)
2. BWS-3  Run Campaign & Measure      WSJF 3.0  M  16h  To Do
3. BWS-4  Build Enrichment Skills     WSJF 1.4  L  40h  To Do
```

---

### Step 3 — Check Google Calendar for Conflicts

**Input:** Query Google Calendar for today (date in HKT).

**Extract:**
- All events between 7 AM and 4 PM
- Include all-day events and events >30 min duration
- Exclude calendar named "Admin" (external calendar, ignored)
- Extract: event name, time, duration

**Output:**
- If no conflicts: "No calendar conflicts detected." → proceed to Step 5
- If conflicts: list conflicts by time → proceed to Step 4

**Note:** Do not fail on calendar API errors; log and continue with no adjustments.

---

### Step 4 — Detect Disruptions & Ask for Adjustment

**Trigger:** Only runs if Step 3 found conflicts.

**Display:** Show calendar conflicts to user:
```
Calendar Conflicts Detected:
- 10:00–11:00 AM: Networking meeting (1h)
  → Reduces available morning focus from 5h to 4h
```

**Ask user:** "Do we need to adjust your ideal day? (yes/no)"

**If yes:**
- Ask: "Describe the change. (e.g., '90-min meeting cuts morning to 3.5h')"
- Update available_focus_hours for Step 5

**If no:**
- Assume you'll absorb the meeting and stay on track
- Use original capacity (7h or 8h)

**Output:** adjusted_focus_hours (or original if no adjustment).

---

### Step 5 — Map Projects to Time Blocks

**Algorithm:**

1. **Start with adjusted capacity from Step 1:**
   - Use adjusted_capacity_hours from Step 1 (8h, 7h, 6h, etc.)
   - Further adjust if calendar conflicts were detected in Step 4
   - Preserve morning block start time from Step 1 (7 AM or 8 AM or later if needed)

2. **Block assignments depend on disruptions:**
   - **Standard (no disruptions, 8h):**
     - Morning block: 7 AM–12 PM = 5 hours
     - Afternoon block: 1–4 PM = 3 hours

   - **Running (7h):**
     - Morning block: 8 AM–12 PM = 4 hours
     - Afternoon block: 1–4 PM = 3 hours

   - **Gym + other disruptions (6h or less):**
     - Recalculate morning block based on start time and duration
     - Afternoon remains 1–4 PM = 3 hours (unless disruption affects afternoon)

3. **Project allocation (in merged order from Step 2):**
   - In Progress stories first — allocate their remaining hours to the top of the morning block
   - Then To Do stories in WSJF order
   - Fit into available blocks: morning first, then afternoon
   - If project spans blocks, split: "2h morning / 6h afternoon"
   - Mark as "allocated" or "partial"
   - If hours exceed remaining capacity, mark as "queued_tomorrow"
   - Stop when capacity is exhausted

4. **Example (7h day with running, 1 story in progress):**
   ```
   BWS-2 (In Progress, 5h remaining):
     - 8–12 PM: 4h allocated ✓
     - 1–2 PM:  1h allocated ✓
     → Total: 5h

   BWS-3 (To Do, WSJF 3.0, 16h):
     - 2–4 PM: 2h allocated (partial)
     → Remaining 14h queued for tomorrow
   ```

5. **Output:** For each story:
   - story_key, story_title, wsjf, status
   - morning_hours, afternoon_hours, total_today_hours
   - allocation_status (allocated/partial/queued_tomorrow)

### Step 6 — Create/Update Google Calendar Events

**For each allocated story:**

**Event Details:**
- **Title:** `[WSJF X.X] Story Title (Yh/8h) · BWS-N`
  - Example: `[WSJF 4.5] Integrate Platform & Send (8/8h) · BWS-2`
- **Time:** As calculated in Step 5
- **Color:** Graphite (default); Tomato if partial allocation or In Progress carry-over
- **Description:**
  ```
  Project: Integrate Platform & Send Campaigns
  WSJF: 4.5 | Jira: BWS-2
  Total Hours: 8 | Scheduled Today: 8h
  Status: In Progress
  Focus Goal: Complete platform integration & launch campaign
  ```

**Calendar Operations:**
- **Create** new events for stories not yet on calendar today
- **Update** existing events if created yesterday with "pending review"
- **Delete** placeholder events marked as "draft"
- **Preserve** existing personal events (calls, breaks, etc.)

**Fallback:** If calendar API fails, log error and display plan as text summary.

## Edge Cases

**No active sprint:** Display warning, fall back to wiki WSJF queue.

**All sprint stories Done:** Congratulate, skip to disruptions check, show empty plan or wiki backlog if any queued projects exist.

**In Progress story has no hours remaining estimate:** Allocate a default 1h focus block and flag: "BWS-X has no hours estimate — defaulting to 1h. Update in Jira if needed."

**No disruptions, no calendar conflicts:** Straight allocation of 8h to top stories.

**Multiple disruptions:** Sum times, ask for clarification if unclear, recalculate capacity.

**Calendar API unavailable:** Display plan as text summary, continue without writing to calendar.

**Jira API unavailable:** Log error, fall back to wiki WSJF queue, flag in summary: "⚠️ Jira unavailable — using wiki queue (may be stale)."

## Notes

1. **Jira is primary, wiki is secondary.** Step 0 fetches live sprint status from Jira.
   The wiki provides WSJF scores only. If the two conflict on hours, Jira story points
   take precedence.

2. **In Progress stories always go first.** Finish what's started before pulling new
   work. This prevents context-switching debt from accumulating across days.

3. **Done stories are excluded.** No need to plan work that's already complete.
   The sprint summary in Step 7 shows Done count for context.

4. **Disruptions are proactively captured.** The skill asks broadly — running, gym,
   lunch meetings, or other deviations. The answer determines adjusted focus capacity.

5. **Multiple disruptions are additive.** Running (1h) + lunch meeting (1h) = 6h
   capacity. If unclear, the skill asks for confirmation.

6. **Calendar conflicts trigger a decision.** If a meeting appears overnight, the
   skill detects it and asks before adjusting. It never silently re-maps projects.

7. **Capacity is a hard limit.** Projects exceeding adjusted capacity are queued
   for tomorrow. The skill never over-commits the day.

8. **Google Calendar is the output.** Your calendar IS your daily plan. Review and
   adjust in Google Calendar before standup.
