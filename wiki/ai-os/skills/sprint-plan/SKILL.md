# Sprint Planning Skill

Runs the sprint planning ceremony. Reads the backlog, guides a prioritisation
discussion, creates the Jira sprint, assigns stories, and hands off for manual
review. The user always starts the sprint — Claude never does.

## Connection Details

- **Site:** agileict.atlassian.net
- **Cloud ID:** `bbbb75d2-e2e4-44fe-a329-e506d1128c29`
- **Board ID:** BWS (Agile Sprints)
- **Sprint capacity:** Velocity-based (see below)

## Capacity — Velocity

Sprint capacity is measured in story points, determined by velocity:

- **Sprints 1–3 (bootstrapping):** Use 10 SP as a starting capacity (roughly
  equivalent to the old 40h anchor). Adjust based on what actually completes.
- **Sprint 4+:** Use rolling average velocity from the last 3 sprints. Read
  `wiki/ai-os/logs/retro-log.md` for historical velocity data.

Display the capacity source at the start of planning:
```
Sprint capacity: [N] SP (based on [rolling 3-sprint avg / bootstrap estimate])
```

## Sizing Reference

| T-shirt | Hours (anchor) | Story Points |
|---------|---------------|-------------|
| XS | 1–4h | 1 |
| S | 5–8h | 2 |
| M | 9–16h | 3 |
| L | 17–40h | 5 |
| XL | 40h+ | 8 — must be split before committing |

## Step-by-Step Process

### Step 1 — Read the backlog

Fetch all unstarted, unsprinted BWS stories using JQL:

```
project = BWS AND issuetype = Story AND status = "To Do" AND sprint is EMPTY ORDER BY priority ASC, created ASC
```

Use `searchJiraIssuesUsingJql`. Extract for each story:
- Key (BWS-N)
- Summary (title)
- Priority (Highest/High/Medium/Low/Lowest)
- Story Points (`customfield_10016`)
- Labels (to extract `wsjf-X.X` and size label)
- Parent Epic summary

If no stories are found, tell the user the backlog is empty and stop.

### Step 2 — Present the ranked table

Display a ranked table sorted by Priority (Highest first), then story points ascending within the same priority. Show a running capacity total as stories are added hypothetically:

```
Sprint capacity: [N] SP (rolling 3-sprint avg / bootstrap)

 #  Key    Story                              Priority   Size   SP   Running
 1  BWS-2  Integrate Platform & Send          High       S      2    2 SP  ✓
 2  BWS-3  Run Campaign & Measure             Medium     M      3    5 SP  ✓
 3  BWS-4  Build Enrichment & Personalisation Low        L      5    10 SP ⚠ over capacity

Stories 1–2 fit within [N] SP (5 SP committed, [N-5] SP remaining).
Story 3 (BWS-4, 5 SP) would exceed capacity.
```

Use ✓ for stories that fit, ⚠ for stories that would exceed capacity.

After displaying the table, say:
"This is your WSJF-ranked backlog. What would you like to commit to this sprint? I'd suggest starting with the top-ranked items — does this order look right to you?"

### Step 3 — Guided discussion

Work through the backlog interactively. Your role is:

**Challenge over-commitment:** If the user wants to add a story that would exceed capacity, flag it:
"Adding [story] would take you to [X] SP — [Y] SP over your velocity of [N]. Which story would you drop to make room, or do you want to reduce your commitment on [large story]?"

**Flag L stories (5 SP):** A 5-point story may consume most of the sprint capacity:
"[Story] is 5 SP. Based on your velocity of [N], that's [X]% of your sprint capacity — are you confident you'll complete it this week, or would you prefer to define a specific sprint goal for just part of it?"

If the user wants to partially commit to an L story, ask:
"What specifically will you complete from [story] this sprint?" Create that as a sub-task in Jira (see Step 4b) and add the sub-task to the sprint instead of the parent story.

**Flag mood-driven re-ordering:** If the user wants to pull a lower-priority story above a higher-priority one, ask:
"[Higher story] has a higher WSJF score — what's the reason for prioritising [lower story] above it this week? If there's a cost-of-delay reason, we can adjust."

**Definition of Done gate (before committing any story):**
For each story the user wants to commit, ask:
"What specifically will done look like for [story]? Give me one sentence that's binary — something you can answer yes or no to."

If the answer is vague (e.g. "improve the enrichment pipeline"), push back:
"That doesn't have a stopping point — what specifically will be true when this story is done? E.g. 'The script processes 100 contacts end-to-end without manual intervention.'"

Only proceed to commit the story once the DoD is specific and testable. Write the confirmed DoD as a comment on the Jira story via `addCommentToJiraIssue`:
`"Definition of Done (sprint planning [date]): [DoD statement]"`

This creates an audit trail and a reference point for the retro.

**Confirm the final list:** Once the user is satisfied, summarise:
```
Sprint commitment:
  BWS-2  Integrate Platform & Send Campaigns   2 SP
         DoD: Platform integrated; first campaign sent to 50 contacts ✓
  BWS-3  Run Campaign & Measure               3 SP
         DoD: Campaign results reported; reply rate and open rate captured ✓
  ─────────────────────────────────────────────
  Total committed: 5 SP / [N] SP capacity ([N-5] SP remaining)
```

Ask: "Happy with this? I'll create the sprint and assign these stories — you'll review in Jira before starting."

### Step 4a — Create the sprint

Use the Jira Agile REST API via `fetch` to create the sprint:

**Endpoint:** `POST https://api.atlassian.com/ex/jira/bbbb75d2-e2e4-44fe-a329-e506d1128c29/rest/agile/1.0/sprint`

**Body:**
```json
{
  "name": "Sprint [N] — [start date] to [end date]",
  "originBoardId": [board ID],
  "startDate": "[YYYY-MM-DDT00:00:00.000Z]",
  "endDate": "[YYYY-MM-DDT23:59:59.000Z]"
}
```

Sprint naming convention: `Sprint 1 — 7 May to 13 May 2026`
Sprint duration: 7 days from today (Mon–Sun or the next 7 calendar days).

To get the board ID, fetch:
`GET https://api.atlassian.com/ex/jira/bbbb75d2-e2e4-44fe-a329-e506d1128c29/rest/agile/1.0/board?projectKeyOrId=BWS`

The sprint number (Sprint N) is one more than the count of existing sprints. Fetch existing sprints to determine N:
`GET https://api.atlassian.com/ex/jira/bbbb75d2-e2e4-44fe-a329-e506d1128c29/rest/agile/1.0/board/{boardId}/sprint`

### Step 4b — Handle partial L story commitments (if applicable)

If the user committed to only part of an L story, create a sub-task under the parent story using `createJiraIssue`:
- `issueTypeName`: "Subtask"
- `parent`: the L story key
- `summary`: the specific sprint goal the user described
- `story_points` / `customfield_10016`: the estimated story points for this sub-task

Add the sub-task key to the sprint assignment list instead of the parent.

### Step 5 — Assign stories to the sprint

For each committed story (or sub-task), use `editJiraIssue` to set the sprint field:

```json
{ "customfield_10020": [sprint_id] }
```

Where `sprint_id` is the ID returned from the sprint creation in Step 4a.

If `editJiraIssue` rejects `customfield_10020`, use the Agile API instead:

**Endpoint:** `POST https://api.atlassian.com/ex/jira/bbbb75d2-e2e4-44fe-a329-e506d1128c29/rest/agile/1.0/sprint/{sprintId}/issue`

**Body:**
```json
{ "issues": ["BWS-2", "BWS-3"] }
```

### Step 6 — Present the sprint summary

```
Sprint created: Sprint 1 — 7 May to 13 May 2026
──────────────────────────────────────────────────
  BWS-2  Integrate Platform & Send Campaigns   2 SP   High
  BWS-3  Run Campaign & Measure               3 SP   Medium
──────────────────────────────────────────────────
  Committed: 5 SP / [N] SP capacity  ([N-5] SP remaining)

Parked this sprint:
  BWS-4  Build Enrichment & Personalisation   5 SP  Low  (over capacity)

Next step: Review the sprint in Jira, make any adjustments, then start it manually.
Board: https://agileict.atlassian.net/jira/software/projects/BWS/boards
```

### Step 7 — Hand off

Tell the user:
"The sprint is created and stories are assigned — but not started. Open the board, review the layout, and start the sprint when you're ready. Any last-minute changes (swapping stories, adjusting priorities) can be made directly in Jira before you hit Start."

**Claude never starts the sprint.** This is always a manual user action.

## Edge Cases

- **No board ID found:** Tell the user to check agileict.atlassian.net and confirm BWS is a scrum board, not Kanban.
- **Sprint creation fails:** Log the error. Offer to assign stories to an existing sprint if one exists, or suggest the user creates the sprint manually in Jira and re-runs `/jira-sync`.
- **Story has no story points set:** Flag it — "BWS-X has no story points estimate. What's your estimate?" Update the story's story points before adding to the sprint.
- **All stories over capacity:** Present the table, note that the smallest story already exceeds capacity, and ask the user which story to commit to partially.
- **XL story in backlog:** Flag immediately — "BWS-X is XL (8 SP) and needs to be split before sprint planning. Would you like to define sub-tasks for it now?"
